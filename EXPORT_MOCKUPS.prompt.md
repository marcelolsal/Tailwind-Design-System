## What This Skill Does

Produces a ZIP file containing all mockup HTML files for a feature, ready to hand directly to a developer or drop into any LLM session that has the mount-mockups skill active. The receiving session only needs to see the ZIP — no instructions required.

---

## Naming Conventions

### ZIP filename
Name the zip after the feature, using kebab-case. This becomes the folder path on the receiving end:

```
discount-flow.zip       → mockups/discount-flow/
checkout-v2.zip         → mockups/checkout-v2/
order-details.zip       → mockups/order-details/
```

A well-named zip means the receiver never has to answer "what feature is this?" — the mount skill reads the filename automatically.

### Mockup filenames
Each HTML file inside the zip must follow this pattern:

```
Mockup NN - Description.html
```

- `NN` is a zero-padded two-digit sequence number (`01`, `02`, … `18`)
- Variants of a screen use a letter suffix: `04B`, `04C`
- Description is title case, plain English, no special characters
- Examples: `Mockup 01 - Full page.html`, `Mockup 04B - Picker modals.html`

## Export Steps

### Step 1 — Generate standalone HTML files

Each mockup screen must be exported as a **standalone bundled HTML file** using `super_inline_html`. This inlines all dependencies (`styles.css`, `_ds_bundle.js`, fonts) into a single file that opens in any browser with no external requests.

**Requirements for each file:**
- Opens in any browser offline
- Loads this design system's runtime inlined: `styles.css` + `_ds_bundle.js` (namespace `window.SzumTechDesignSystem_1f5ec4`) — not linked externally
- Includes the Poppins / JetBrains Mono webfonts inlined as base64 (not a live Google Fonts request)
- Has a `<template id="__bundler_thumbnail">` with a simple SVG splash
- Renders correctly at 100% zoom in a standard browser window
- **Preserves any `<!-- @prdScreen <slug> -->` identity marker verbatim.** If the source file (generated by `implement-prd`) has this comment near the top, it must survive inlining/minification untouched — bundlers commonly strip HTML comments. After bundling, grep the output file for `@prdScreen` before proceeding. If a file had the marker before bundling and doesn't have it after, the bundler stripped it — fix this before zipping, don't ship it silently. Files that never had a marker (hand-built mockups, not PRD-generated) don't need one.

### Step 2 — Name and sequence the files

Number the screens in the logical user flow, not alphabetically or by build order. The sequence should tell the story:

1. The empty/default state
2. Each section of the form in order
3. Interactive/picker states (lettered variants)
4. Save, loading, success, error states
5. Edit mode screens
6. Delete / destructive action screens

**Gaps in numbering are expected, not a defect, when a screen was deliberately retired.** If these mockups came from `implement-prd`, cross-check the sequence against the Design-PRD's own Screen Inventory rather than assuming every gap is a mistake — `implement-prd` intentionally does not renumber stable screens just because another screen was removed.

### Step 2B — Include the Design-PRD

If these mockups were generated from a Design-PRD (via `implement-prd`), include the **actual Design-PRD document** — or at minimum its Screen Inventory and Per-Screen Specs sections — in the zip root as `design-prd.md`. This is not a summary you write yourself; copy the real document. It travels with the mockups so the implementing developer has the exact states, copy, labels, and reuse provenance as ground truth, instead of reverse-engineering intent from static HTML alone.

If there is no Design-PRD (a hand-built or ad-hoc mockup), skip this step — there's nothing to include.

### Step 3 — Create the ZIP

Use `run_script` to create the zip:

```js
// Collect all standalone HTML files
const files = [
  'Mockup 01 - Full page.html',
  'Mockup 02 - Amount off products.html',
  // …
];

// Minimal ZIP writer (stored, no compression — HTML is already large text)
function zipStored(entries) {
  const encoder = new TextEncoder();
  const localHeaders = [];
  const centralDir = [];
  let offset = 0;

  for (const { name, data } of entries) {
    const nameBytes = encoder.encode(name);
    const dataBytes = typeof data === 'string' ? encoder.encode(data) : data;
    const crc = crc32(dataBytes);
    const size = dataBytes.length;

    const lh = new Uint8Array(30 + nameBytes.length);
    const lhv = new DataView(lh.buffer);
    lhv.setUint32(0,  0x04034b50, true);
    lhv.setUint16(4,  20, true);
    lhv.setUint16(6,  0, true);
    lhv.setUint16(8,  0, true);
    lhv.setUint16(10, 0, true);
    lhv.setUint16(12, 0, true);
    lhv.setUint32(14, crc, true);
    lhv.setUint32(18, size, true);
    lhv.setUint32(22, size, true);
    lhv.setUint16(26, nameBytes.length, true);
    lhv.setUint16(28, 0, true);
    lh.set(nameBytes, 30);

    localHeaders.push(lh, dataBytes);

    const cd = new Uint8Array(46 + nameBytes.length);
    const cdv = new DataView(cd.buffer);
    cdv.setUint32(0,  0x02014b50, true);
    cdv.setUint16(4,  20, true);
    cdv.setUint16(6,  20, true);
    cdv.setUint16(8,  0, true);
    cdv.setUint16(10, 0, true);
    cdv.setUint16(12, 0, true);
    cdv.setUint16(14, 0, true);
    cdv.setUint32(16, crc, true);
    cdv.setUint32(20, size, true);
    cdv.setUint32(24, size, true);
    cdv.setUint16(28, nameBytes.length, true);
    cdv.setUint16(30, 0, true);
    cdv.setUint16(32, 0, true);
    cdv.setUint16(34, 0, true);
    cdv.setUint16(36, 0, true);
    cdv.setUint32(38, 0, true);
    cdv.setUint32(42, offset, true);
    cd.set(nameBytes, 46);
    centralDir.push(cd);

    offset += lh.length + dataBytes.length;
  }

  const cdOffset = offset;
  const cdBytes = concat(...centralDir);
  const eocd = new Uint8Array(22);
  const ev = new DataView(eocd.buffer);
  ev.setUint32(0, 0x06054b50, true);
  ev.setUint16(4, 0, true);
  ev.setUint16(6, 0, true);
  ev.setUint16(8, entries.length, true);
  ev.setUint16(10, entries.length, true);
  ev.setUint32(12, cdBytes.length, true);
  ev.setUint32(16, cdOffset, true);
  ev.setUint16(20, 0, true);

  return concat(...localHeaders, cdBytes, eocd);
}

function concat(...arrays) {
  const total = arrays.reduce((s, a) => s + a.length, 0);
  const out = new Uint8Array(total);
  let offset = 0;
  for (const a of arrays) { out.set(a, offset); offset += a.length; }
  return out;
}

function crc32(data) {
  let crc = 0xFFFFFFFF;
  const table = crc32.table || (crc32.table = (() => {
    const t = new Uint32Array(256);
    for (let i = 0; i < 256; i++) {
      let c = i;
      for (let j = 0; j < 8; j++) c = (c & 1) ? (0xEDB88320 ^ (c >>> 1)) : (c >>> 1);
      t[i] = c;
    }
    return t;
  })());
  for (const byte of data) crc = table[(crc ^ byte) & 0xFF] ^ (crc >>> 8);
  return (crc ^ 0xFFFFFFFF) >>> 0;
}

// Build entries
const entries = [];
for (const name of files) {
  const html = await readFile(`<source-path>/${name}`);
  entries.push({ name, data: html });
}
// Include the Design-PRD if this feature came from implement-prd (Step 2B)
if (await fileExists('<source-path>/design-prd.md')) {
  entries.push({ name: 'design-prd.md', data: await readFile('<source-path>/design-prd.md') });
}

const zipBytes = zipStored(entries);
const zipBlob = new Blob([zipBytes], { type: 'application/zip' });
await saveFile('export/discount-flow.zip', zipBlob);
log('ZIP created: export/discount-flow.zip');
```

### Step 4 — Deliver

Call `present_fs_item_for_download` with the zip path so the designer can download it and send it to their developer directly.

```
Here are the mockups for [Feature Name].

File: [feature-slug].zip
Screens: N standalone HTML files

each screen so you're not guessing from the HTML alone.

To view: mount the zip in your Claude session. Just attach the file and say "mount mockups".
To implement: after mounting, say "implement this mockup" on any screen.
```

## Quality Checks Before Exporting

Run these checks on each HTML file before zipping:

| Check | How |
|---|---|
| Opens offline | Open in browser with network disabled — no blank/broken state |
| Bundler splash present | File has `<div id="__bundler_thumbnail">` |
| Correct runtime | `styles.css` + `_ds_bundle.js` are inlined (not external links/CDN references) |
| Fonts inlined | No external Google Fonts or CDN font requests |
| Correct filename pattern | `Mockup NN - Description.html` |
| Sequence stable, not necessarily dense | Gaps are fine if the Design-PRD's Screen Inventory shows the screen was deliberately removed; a gap with no corresponding removal is a mistake |
| Identity marker intact | If the source file had `@prdScreen ...`, grep the bundled output — don't assume it survived inlining |
| PRD included when applicable | `design-prd.md` is present at the zip root for any feature generated via `implement-prd` |

If any check fails, re-export the offending file with `super_inline_html` before zipping.

## File Layout Inside the ZIP

```
discount-flow.zip
  Mockup 01 - Full page.html
  Mockup 02 - Amount off products.html
  Mockup 03 - Discount value.html
  …
  Mockup 18 - Delete loading and error.html
  design-prd.md              ← only if this feature came from implement-prd
```

No subfolders inside the zip. All HTML files at the root level. The zip name is the feature name.
